# External Design Lessons

Use this note when the user asks for broader craft improvement beyond the source deck and internal references.

After choosing a visual direction, pick one or two sources from this list and write the concrete transferable rules into `deck-design-system.md` or `deck-design-system.json`. Record rules, not mood-board impressions.

## Transferable Lessons

### 1. Story Before Styling

- Pitch's template workflow starts with audience, purpose, and outline before visual exploration. Keep that order in deck rebuilds: story structure and reusable slide jobs should be decided in plain text before composition styling.
- When a deck is weak, do not start by decorating it. First rewrite the outline, cut redundancy, and decide which slide types actually carry the argument.

### 2. Deck Systems Beat Per-Slide Decoration

- Microsoft and Apple both treat slide layouts as reusable systems that can be edited and reapplied across many slides. Translate that into web work by defining a deck shell, title band, archetypes, and layout patterns before tuning individual pages.
- If a page only works because of one-off local overrides, the deck system is still under-specified.

### 3. Titles Should Clarify, Not Perform

- Use typography to preserve glanceability and reading order. Do not create drama by forcing narrow title boxes or ornamental line breaks.
- Fix title posture with width, copy, and hierarchy before shrinking the type.
- Smaller text and tertiary labels should stay secondary. Do not shrink core headings or key conclusions just to preserve a decorative composition.

### 4. Constrained Spaces Need Stronger Editing

- Pitch explicitly studies how information behaves in restricted spaces. Dense slides need stronger selection, not just smaller type.
- Images and decorative shapes should stay only when they add informational or rhetorical value. Remove visuals that lengthen the page without improving understanding.
- Spacious pages still need a held composition. A floating title above disconnected cards is usually a planning failure, not sophistication.

### 5. One Orientation Device Is Usually Enough

- Offer one clear deck overview or navigation system. Repeating a second contents page is rarely useful unless it reframes the story in a new way.
- Once the opening shell has oriented the viewer, the next slide should deepen stakes, evidence, or decision context.

### 6. Feedback Should Target The Why

- Good presentation feedback explains why a direction works or fails, not just which variant looks nicer.
- During QA, explicitly mark pages that feel empty, duplicated, overexplained, or generic and redesign them before scaling the system to the full deck.

## How To Apply Them To Deck Rebuilds

- Start external learning by asking: what is the deck for, who is it persuading, and which slide jobs must exist.
- Convert inspiration into system rules: shell, title posture, density target, image policy, proof devices, and anti-patterns.
- First fix heading width and line-break behavior before fine-tuning font size.
- Remove repeated agenda pages when cover navigation already orients the viewer.
- Let diagrams, timelines, and comparison structures carry the main argument; use copy only to interpret or sharpen the takeaway.
